## 已验证结论

关于最新版 PR `#38479`，现在已经可以把结论拆成两层。

第一层是“源码 build 和 TurboQuant runtime 是否整体可用”。这部分结论是肯定的。本机 `zgp-vllm-pr38479-snap:latest` 快照镜像虽然默认带的是旧 PR 的 editable `vllm`，但它有足够的编译工具链和 CUDA 开发库，最新版 PR 已经在这套镜像里完成过一次完整源码编译。随后通过源码模式最小化运行也证明：只要绕过 package metadata / 平台探测这类外围问题，最新版 PR 的 TurboQuant 运行链路在 A100 上并不是整体失效。

第二层是“为什么 `Qwen3-32B-FP8` 在 A100 上跑不起来”。这部分结论也已经明确：真正的阻塞不在 TurboQuant attention backend，而在 block-FP8 权重 GEMM kernel。`Qwen3-32B-FP8` 会触发 block-FP8 线性层 kernel 选择，A100 上可用候选实际只剩 `TritonFp8BlockScaledMMKernel`，而它运行时会在 Triton 编译阶段报 `fp8e4nv` 类型不受当前架构支持。把 Triton 禁掉后，其它候选 `FlashInfer/DeepGemm/Cutlass` 又都不支持 `sm80`。因此，最新版 PR 在 A100 上对这个模型失败，是上游 block-FP8 权重 kernel 覆盖缺口，不是 TurboQuant KV-cache 路径本身坏掉。

同时还用 `/share/models/official/Qwen3-8B` 做了一次最小分离验证。这个模型不带 FP8 权重，因此不走 `Qwen3-32B-FP8` 那条 block-FP8 权重路径。结果是最新版 PR 能在 A100 上顺利完成：

- 边界层 `FLASH_ATTN` + 中间层 `TURBOQUANT` 的 backend 选择；
- 权重加载；
- `torch.compile`；
- mixed prefill/decode CUDA graph capture；
- 最终 one-off `LLM.generate()`。

这次运行中测得 `GPU KV cache size: 437,664 tokens`，在 `max_model_len=4096` 下的 `Maximum concurrency` 约为 `106.85x`。这说明最新版 PR 的 TurboQuant runtime 本身在 A100 上是能工作的，只是 `Qwen3-32B-FP8` 这类 block-FP8 权重模型当前超出了 A100 上游 kernel 覆盖范围。

## 迁移到 H200 的建议路径

H200 迁移的核心价值在于，它天然更接近当前 PR 依赖的 block-FP8 kernel 目标平台。基于本轮 A100 结论，建议在 H200 上优先按下面顺序推进，而不是重复 A100 上已经证伪的问题。

首先，直接复现 `Qwen3-32B-FP8` 的最新版 PR smoke。A100 上真正卡住的是 block-FP8 权重 kernel，因此 H200 是最应该先验证的环境。如果 H200 能让 `Qwen3-32B-FP8` 跑通，那么 PR 方案与插件方案的 apples-to-apples 对比才有意义。

其次，再补 PR vs 插件的 serving benchmark 与能力对比。建议继续沿用 `Qwen3-32B-FP8`、`/v1/completions`、`lm_eval local-completions` 这套契约，这样可以直接接上 [19-pr-latest-vs-plugin-qwen3-2026-04-15.md](/ceph/User/E01442/turboquant/doc/turboquant-pr38479-qwen35-benchmark/19-pr-latest-vs-plugin-qwen3-2026-04-15.md) 里的设计，而不是重新设计整套评测协议。

最后，如果 H200 上 `Qwen3-32B-FP8` 仍然失败，再把问题重新压回到 block-FP8 kernel 支持矩阵本身。届时应优先检查 `FlashInfer/DeepGemm/Cutlass/Triton` 中哪个 kernel 被选中、报错是否仍与 `fp8e4nv` 类型有关，而不是再重复追查 `LD_LIBRARY_PATH`、源码 build 或 TurboQuant layer skip 之类已经在 A100 上排干净的问题。

## 建议直接复用的命令

在 H200 上复现最新版 PR 的核心 smoke，建议不要直接用 `vllm` CLI 主入口，而是优先沿用本轮已经验证过的“源码 build 根目录 + 最小 metadata”方式做 one-off `LLM` 验证。A100 上成功跑通 `Qwen3-8B` 的思路如下：

```bash
export LD_LIBRARY_PATH=/usr/local/nvidia/lib64:/lib/x86_64-linux-gnu
python3 -m pip uninstall -y vllm

mkdir -p /tmp/prmeta/vllm-0.19.1rc1.dev301.dist-info
cat > /tmp/prmeta/vllm-0.19.1rc1.dev301.dist-info/METADATA <<'EOF'
Metadata-Version: 2.1
Name: vllm
Version: 0.19.1rc1.dev301+gac46a983e.d20260416
EOF

export PYTHONPATH=/path/to/pr38479_upstream_build_root:/tmp/prmeta
export VLLM_TARGET_DEVICE=cuda
```

然后在同一个 Python 进程里显式让 `arg_utils` 绑定到实际的 CUDA 平台对象，再调用：

```python
from vllm import LLM, SamplingParams
llm = LLM(
    model="/share/models/official/Qwen3-32B-FP8",
    trust_remote_code=True,
    max_model_len=4096,
    gpu_memory_utilization=0.55,
    kv_cache_dtype="turboquant_4bit_nc",
)
```

如果是 API server 验证，则不要显式传 `--attention-backend TURBOQUANT`。最新版 PR 会自动把首尾 2 层加入 `kv_cache_dtype_skip_layers`；这些边界层需要回退到普通 backend。只传 `--kv-cache-dtype turboquant_4bit_nc`，让中间层自动选 `TURBOQUANT`、边界层自动退回 `FLASH_ATTN` 才是正确路径。
