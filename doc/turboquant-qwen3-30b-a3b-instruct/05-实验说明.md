# Qwen3-30B-A3B-Instruct 上 TurboQuant 实验说明

## 1. 一句话结论

在 `Qwen3-30B-A3B-Instruct-2507 + 2xA100 80GB + TP=2` 上，TurboQuant 已经能把
当前请求对应的 **active KV occupancy** 从 `102,088,704 bytes/rank` 压到
`20,470,912 bytes/rank`，约 `4.99x`。

## 2. 实验环境

- 模型：
  `/share/models/official/Qwen3-30B-A3B-Instruct-2507`
- GPU：
  `CUDA_VISIBLE_DEVICES=0,1`
- 并行：
  `TP=2`
- 软件：
  - `vllm==0.17.0`
  - `torch==2.10.0+cu129`
  - 本地 `turboquant`

模型关键信息：

- 架构：`Qwen3MoeForCausalLM`
- 层数：`48`
- KV heads：`4`
- `head_dim=128`

这个模型的重要特点是：TurboQuant 最终能 hook **全部 48 个 attention 层**。

## 3. 这个结论是怎么跑出来的

核心命令是：

```bash
CUDA_VISIBLE_DEVICES=0,1 \
VLLM_ENABLE_V1_MULTIPROCESSING=0 \
MODEL=/share/models/official/Qwen3-30B-A3B-Instruct-2507 \
TP=2 \
MAX_MODEL_LEN=8192 \
GPU_MEM=0.85 \
INPUT_TOKENS=2048 \
CONCURRENCY=1 \
MAX_TOKENS=16 \
MODE=accumulate \
vllm/.venv/bin/python \
doc/turboquant-qwen3-30b-a3b-instruct/task-kv-occupancy/inspect_kv_occupancy.py
```

这个脚本做了三件事：

1. 构造一条长历史 prompt
2. 跑一次真实生成，让 KV 真正写入
3. 分别统计：
   - baseline active KV 占用
   - TurboQuant 压缩后的 active KV 占用

这里主口径只看 **active occupancy**，不看整块预分配 backing storage。  
原因是 vLLM 会预留很大的 KV 缓冲，如果拿整块 backing storage 来算，会把压缩比严重夸大。

## 4. `102,088,704 -> 20,470,912` 是怎么计算的

### 4.1 baseline：`102,088,704 bytes/rank`

这个数来自公式：

```text
seq_len × layer_count × num_kv_heads_per_rank × head_dim × 2 × dtype_bytes
```

本次实验实际取值：

- `seq_len = 2077`
  - 这是 tokenizer 后真实长度，不是目标值 `2048`
- `layer_count = 48`
- `num_kv_heads_per_rank = 2`
  - 模型总共 `4` 个 KV heads，`TP=2`，每个 rank 上是 `2`
- `head_dim = 128`
- `2`
  - 表示 `K + V`
- `dtype_bytes = 2`
  - baseline 用 `bfloat16`

所以：

```text
2077 × 48 × 2 × 128 × 2 × 2
= 102,088,704 bytes/rank
```

### 4.2 TurboQuant：`20,470,912 bytes/rank`

这个数不是公式估的，而是直接把每个 `TurboQuantKVCache.memory_bytes()`
返回值加总得到的：

- `quantized_keys = 10,900,096`
- `quantized_values = 9,570,816`
- `buffer = 0`

所以：

```text
10,900,096 + 9,570,816 = 20,470,912 bytes/rank
```

### 4.3 压缩比

```text
102,088,704 / 20,470,912 ≈ 4.99
```

所以当前最准确的表述是：

> 在这次 `~2k token, TP=2, concurrency=1` 的实验点上，TurboQuant 将 active KV occupancy 从 `102,088,704 bytes/rank` 压到 `20,470,912 bytes/rank`，压缩比约 `4.99x`。

## 5. 当前 demo 的边界

当前这版 demo 可以诚实地说：

1. TurboQuant 在 `Qwen3-30B-A3B-Instruct-2507` 上已经实现了 KV cache 压缩
2. active KV occupancy 在当前实验点上约 `4.99x`

但它还不能直接说成：

- 已经是成熟的大规模在线服务方案

原因很简单：

- 当前实现是 **layer hook + side-cache**
- 不是 **request-aware / scheduler-aware 的 cache engine**

也就是说，它更像：

- 一个已经能工作的 TurboQuant 原型

而不是：

- 一个已经深度接入 vLLM 在线服务生命周期的完整 KV 引擎

## 6. 如果后续要大规模应用，需要做什么

如果目标是：

- 多用户
- 多轮长对话
- 单实例持续在线服务

那么后续重点不应该只是继续 patch attention，而是要：

1. 按请求/会话管理压缩后的 KV
2. 深度接入 vLLM 的 KV block / cache manager
3. 和 scheduler、continuous batching、prefix caching 协同
4. 把“压缩 -> 释放 -> 新请求复用 block”做成一个闭环

一句话：

> 当前 demo 已经证明了 TurboQuant 方法本身能压缩 KV cache；  
> 如果要走向大规模在线服务，后续需要深度集成，实质上替换或接管 vLLM 的 cache 引擎。
